# 第 1 章　从 Prompt 到 Harness：Agent 工程范式的演进

> 你以为你在调一个更聪明的模型，其实你在搭一套更可靠的工程系统。

如果你最近才开始做 Agent，大概率经历过这样一条路径：先写一段精巧的 Prompt，让模型「扮演」一个助手；接着把几个函数包装成「工具」塞给它；然后发现模型有时调对、有时调错、有时干脆胡编参数；于是你不停地改 Prompt、加约束、写「如果……就……」的规则……直到某一天你意识到，自己写的早已不是 Prompt，而是一整套**包裹在模型外面的工程系统**。

这套系统，就是本书的主角——**Harness（执行骨架）**。

本章先建立全书的认知地基：什么是 Harness、它由哪些职责组成、为什么同一个模型套上不同 Harness 会长成性格迥异的 Agent。然后我们点明 pi 在这件事上的独特立场——**它把「约束」这件最敏感的事，刻意从核心里拿掉了**。

---

## 1.1 `Agent = Model + Harness`：能力边界由谁决定

先给一个贯穿全书的公式：

```
Agent = Model + Harness
```

- **Model**（模型）：负责「想」——根据上下文生成文本、决定调哪个工具、传什么参数。
- **Harness**（执行骨架）：负责「做」与「管」——把工具真正执行掉、把结果喂回去、决定模型看到多少历史、在出错时兜住、在危险时刹车。

很多人把 Agent 的能力归因于模型。但只要你拆过任何一个生产级 Agent 的代码就会发现：**模型只提供「意图」，能不能、敢不敢、稳不稳地把意图变成现实，全由 Harness 说了算。**

举个最直观的例子。模型说「我要执行 `rm -rf /tmp/cache`」。这只是一段文本。接下来发生什么，完全取决于 Harness：

- 它会不会先把这条命令拿给人审批？
- 它会不会把命令丢进一个只能访问 `/tmp` 的容器里执行？
- 命令卡住了，它会不会在 30 秒后超时杀掉进程树？
- 命令报错了，它会把错误当异常抛出中断整个会话，还是当成一条「工具结果」喂回给模型让它自己想办法？

这四个问题，模型一个都管不了。它们全是 Harness 的职责。**能力边界不由模型的智商决定，由 Harness 的设计决定。**

## 1.2 为什么同一个模型在不同产品里性格迥异

你一定有过这种体感：同样是 Claude，在 A 产品里小心翼翼、每一步都要你确认；在 B 产品里大刀阔斧、几秒钟改完十个文件。模型权重一模一样，差别全在 Harness。

我们可以把这种「性格」拆成几个可观测的工程维度：

| 维度 | 由 Harness 的哪一部分决定 |
|---|---|
| **胆子大小** | 权限/审批策略：动手前是否要确认、哪些操作直接放行 |
| **记性长短** | 上下文工程：保留多少历史、何时压缩、压缩时丢什么 |
| **手脚多少** | 工具系统：挂载了哪些工具、工具的边界硬化程度 |
| **抗挫折力** | 错误恢复：失败是中断会话还是反馈给模型重试 |
| **可被打断性** | 编排与信号：能否中途插话（steering）、能否优雅中止（abort） |

本书要做的，就是顺着 pi 的真实代码，把这几个维度逐一拆开看个透。当你读完，再看任何一个 Agent 产品，都能反推出它的 Harness 是怎么设计的——以及它的「性格」是怎么被工程出来的。

## 1.3 Harness 的六个核心职责

为了让后面的章节有一张「地图」，这里先把 Harness 的职责归纳成六块。这六块也大致对应了本书的部分划分：

1. **上下文（Context）**：决定每次请求时模型看到什么。包括历史消息的组装、系统提示词的构建、超长时的压缩与剪枝。→ 第六部分
2. **工具（Tools）**：把「能做什么」暴露给模型，并把模型的工具调用真正执行掉。包括工具的描述、参数校验、执行后端。→ 第四部分
3. **编排（Orchestration）**：把「生成 → 调工具 → 再生成」这个循环组织成一个能自主推进、能停下、能恢复的状态机。→ 第二部分
4. **状态（State）**：把会话持久化下来，支持恢复、回溯、分支。→ 第六部分
5. **观测（Observability）**：把循环内部发生的一切（流式输出、工具执行、token 消耗）以事件形式暴露出去，供 UI 渲染、日志记录、扩展挂钩。→ 贯穿全书
6. **约束（Constraints）**：在动手前踩刹车——权限、审批、沙箱、危险操作识别。→ 第五部分

这六块缺一不可。但**不同 Agent 对它们的「权重分配」截然不同**——而这正是 pi 最值得玩味的地方。

## 1.4 pi 的独特立场：把「约束」外包给扩展与容器

读到这里你可能以为，一个「生产级」Agent 一定会把上面六块都做得很重，尤其是「约束」——毕竟谁也不想让模型在生产环境里乱删文件。

pi 的选择恰恰相反。

翻开 pi 的 README，你会反复看到一句话的不同表述：**核心不内置权限系统**。pi 不在自己的循环里写「这条命令危险，要不要确认」的逻辑。它把这件事**外包**出去：

- 想要审批？写个扩展（Extension），在 `tool_call` 钩子里拦下来弹窗确认。
- 想要沙箱？把整个 pi 跑进一个 Docker 容器，或者用 OpenShell 这类受限 shell 把 `bash` 工具的执行后端换掉。
- 什么都不想要？那 pi 就真的会直接执行——它默认相信你已经在一个你信任的环境里运行它。

这是一个大胆、甚至有点「危险」的设计决策。它背后的工程哲学是**机制与策略的彻底分离（mechanism / policy separation）**：

> pi 的核心只提供**机制**——一个稳定、精简、可预测的执行骨架；所有**策略**——该不该执行、在哪执行、出错怎么办——都通过钩子和容器下放给使用者。

这种立场带来一组鲜明的取舍：

| 收益 | 代价 |
|---|---|
| 核心代码极精简，循环逻辑容易推理与测试 | 默认不安全，「开箱即用」的安全感为零 |
| 约束策略可以任意组合，不被框架的安全观绑架 | 安全责任转移给使用者，用错了后果自负 |
| 同一个内核能接入容器、远程、CI 等各种环境 | 需要使用者真正理解「该把约束放在哪一层」 |

在本书第五部分，我们会专门用一整章来讨论这个「无权限」哲学——它不是疏忽，而是一个深思熟虑的架构表态。理解了这一点，你才能理解 pi 的所有其他设计：**正因为核心要保持精简到可以「裸奔」，它的每一处接缝都被设计成可替换、可注入、可外包。**

## 1.5 本书方法论：以真实代码为唯一真理来源

最后说一下本书的写法约定，因为它直接影响你该怎么读。

- **代码是唯一真理**。本书的每一个论断，都尽量落到 pi 仓库里一个具体的文件、函数、甚至行号。解读时仓库 HEAD 为 `1da90398`。如果你读到某处觉得「真的吗」，最好的办法是打开对应文件自己看一眼。
- **顺着一次请求走**。我们不按「模块字典」的方式罗列 API，而是跟随一次真实请求的执行链路——从你在终端敲下一句话，到模型生成、工具执行、结果回灌、会话落盘——把沿途经过的每个子系统拆开。
- **每章自带「关键文件清单」与「动手实验」**。前者帮你快速定位源码，后者逼你动手验证，而不是读完就忘。
