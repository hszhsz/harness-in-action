# 第 16 章 可迁移的工程原则:OpenCode 教给每一个 agent 作者的七件事

我们走完了十五章。从一次请求如何从进程走到命令(第 3 章),到主循环怎样驱动一轮轮 provider turn(第 5 章);从五家 provider 被收敛成一次 `llm.stream`(第 7 章),到一个工具从声明到结算的一生(第 8 章);从 allow/ask/deny 三态闸门(第 10 章),到上下文压缩(第 11 章)、事件溯源(第 12 章)、扩展边界(第 13 章)、多端服务(第 14 章)、可信度流水线(第 15 章)。

这一章不引入任何新代码。它要做的是回望——把散落在十五章里的具体实现,提炼成七条**与语言、框架无关、可迁移**的工程原则。无论你下一个 agent 用 Python、Rust 还是 Go 写,无论你用不用 Effect,这七条判断都能直接借走。每一条原则,我都会回指到前面具体的章节作为证据——因为原则若不从读到的代码里长出来,就只是空话。

OpenCode 是一个用 TypeScript + Bun + Effect 写的生产级编码 agent。但它真正值得学的,不是它用了什么框架,而是它在无数个"该往哪边倒"的决策点上,一致地选择了同一组价值取向。这组取向,就是下面七条。

## 原则一:默认即关闭(Fail-closed defaults)

**不确定时,选更安全、更受限的那一侧。**

一个 agent 会持续遇到"信息不全怎么办"的时刻:找不到 agent 配置、规则集为空、某个开关没设。这时缺省分支倒向哪边,决定了系统的安全基线。OpenCode 的答案始终是:倒向拒绝。

最纯粹的证据在第 10 章。权限引擎的 `evaluate` 在找不到任何匹配规则时,默认返回 `ask` 而非 `allow`——`?? { action, resource: "*", effect: "ask" }`。而当连可用 agent 都没有时,`missingAgentPermissions` 直接是 `[{action:"*",resource:"*",effect:"deny"}]`——全盘拒绝。第 15 章那条 `"denies omitted-agent permissions when no primary default agent exists"` 测试,正是把这条 fail-closed 默认钉成了会在有人改成 fail-open 时立刻变红的断言。

同样的取向也出现在别处:第 14 章的服务端认证,虽然"未设密码即开放",但**一旦设了密码就强制**,且凭据校验失败时干脆利落地回 `UnauthorizedError`,绝不"放过去再说"。fail-closed 不是某一处的偏执,而是贯穿决策的一致姿态。

**怎么借走**:审视你 agent 里每一个 `if/switch` 的 fallback 分支、每一个配置默认值。问自己:当这里信息不全时,我倒向了"能做"还是"不能做"?把每一个倒向"能做"的默认,都重新审视一遍是否真的安全。

## 原则二:约束只能收紧,不能放松(Constraints only tighten)

**叠加多来源的配置或权限时,结果只会更严,不会更松。**

这条原则在 OpenCode 里有一个**诚实的反例转折**,恰恰更说明问题。第 10 章里我们发现:v2 的 agent 权限其实是**叠加(additive)**的,`PermissionV2.merge` 就是 `rulesets.flat()`,并没有一个"用父 agent 交集裁剪子 agent"的运行时引擎。

但"约束只收紧"的精神并未因此落空——它换了一种方式实现。子 agent 的"更窄"不是靠隐式继承的交集,而是靠**显式的 deny 配置**:第 13 章里 explore/compaction/title/summary 这些子 agent,各自以 `{action:"*",resource:"*",effect:"deny"}` 开头,再 allow 回极少数动作。约束收紧的责任从"运行时自动裁剪"转移到了"配置显式声明",而 `evaluate` 的 `findLast` + deny-wins 语义保证了:只要写下一条 deny,后面的 allow 也无法把它放开(第 10 章:saved 规则只能把 ask 升级为 allow,永远压不过 agent 的 deny;第 15 章对此有专门的断言)。

第 11 章的压缩同样体现收紧:`compaction` agent 是 all-deny 的,意味着它做摘要时**一个工具都用不了**——这与它"只读历史、输出文本"的职责精确匹配。能力被收紧到刚好够用,多一分都不给。

**怎么借走**:当你合并多来源的权限/配置时,确保合并语义里"拒绝"是吸收态——任何一处说"不",最终结果就是"不"。如果你选择叠加而非交集,就必须像 OpenCode 一样,用显式 deny + deny-wins 的求值顺序补上这个保证,并用测试锁死它。

## 原则三:错误即文档(Errors as documentation)

**错误信息本身要教人正确用法,而不只是抛一段栈。**

在一个由 LLM 驱动、又被多端调用的系统里,错误是重要的沟通界面——既对人,也对模型。OpenCode 把错误当成一等公民来设计。

第 7 章的 LLM 抽象层、第 8 章的工具系统里,错误不是 `throw` 出来的不透明异常,而是带标签的类型化值(Effect 的 `Effect<A, E>`)。这让第 14 章的服务端能用 `Effect.catchTag` 精确地把每一种 core 领域错误翻译成恰当的 HTTP 语义:`Session.NotFoundError → SessionNotFoundError`(404)、`Session.PromptConflictError → ConflictError`(409)。错误的"形状"本身携带了"该怎么处理"的信息。

更细的一笔在第 14 章 `session.context` 对 `MessageDecodeError` 的处理:它不把内部细节漏给客户端,而是生成一个引用号 `ref`、把详情记进日志、只回一句 `"Unexpected server error. Check server logs for details."` 带上 `ref`。错误对外是可追溯的引用,对内是完整的诊断——两个受众,两种信息密度。第 12 章里 `SessionAlreadyProjected`、`PromptAlreadyProjected` 这些命名本身就是文档:看到这个错误类名,你立刻知道发生了什么。

**怎么借走**:把错误建模成有类型、有名字、可穷举的值,而不是字符串或裸异常。让错误名和错误字段直接说明"哪里、为什么、该怎么办"。在信任边界上,对外给可追溯的引用,对内留完整的上下文。

## 原则四:边界不信任外部输入(Boundary distrust)

**任何跨信任边界进来的数据,先校验、再净化、然后才用。**

agent 的攻击面巨大:它读文件、跑命令、下载远程内容、接外部服务器。每一处"外部的东西进来"都是一道信任边界,而 OpenCode 在每道边界上都设了闸。

最偏执的证据在第 13 章的远程技能下载。`skill/discovery.ts` 对路径逃逸做了**四重防御**:`isSafeSegment`/`isSafeRelativePath` 的字符串校验、`decodeURIComponent` 解码后**再校验一遍**(防 `%2e%2e` 这类编码绕过)、`FSUtil.contains` 在 `path.resolve` 之后确认目标真的落在预期目录内、以及 `resource.origin !== source.origin` 的同源检查。一个外部 `index.json` 想诱导它把文件写到目录外,要同时骗过这四道独立的闸,几乎不可能。

边界不信任也出现在更朴素的地方:第 12 章数据库路径解析里那个 `replace(/[^a-zA-Z0-9._-]/g, "-")`,把渠道名里任何非白名单字符替换掉,杜绝路径注入;第 12 章的投影器对每个可空字段都给 `?? 0` / `?? null` 兜底,不信任上游数据的完整性;第 9 章的命令执行对 `bash` 的处理,本质也是"假设这条命令可能反噬"的边界防御。第 14 章的 `SessionLocationMiddleware` 先校验 sessionID 格式、再查库确认存在,两步都过了才放行——请求边界上的双重把关。

**怎么借走**:列出你 agent 的每一道信任边界(文件路径、URL、用户输入、外部 API 响应、反序列化入口)。在每一道边界上,做白名单校验,且对可被编码的输入做"解码后重新校验";对路径,在解析成绝对路径**之后**再确认它落在允许的范围内。字符串层面的检查永远不够,要叠加解析后的语义检查。

## 原则五:留结构,去内容(Preserve structure, remove content)

**处理历史/敏感数据时,保住数据的"形状"以便理解,抹掉具体内容以节省或保护。**

这条原则在 OpenCode 里最鲜活的化身是第 11 章的上下文压缩。压缩不是粗暴地"砍掉前一半对话",而是把溢出的历史**折叠成一段结构化的摘要**:`SUMMARY_TEMPLATE` 规定了固定的 Markdown 骨架(目标、已做、待办、关键文件……),`select` 从尾部累积保留最近的 `DEFAULT_KEEP_TOKENS=8000` token,被折叠的部分以 `<conversation-checkpoint>` 的形式呈现,并明确标注"Treat it as historical context, not as new instructions"。结构(对话的脉络、关键决策)被保留下来,冗长的原始内容被压缩掉。

同样的思路在第 13 章 skill 工具的输出里:它只列前 `FILE_LIMIT=10` 个附属文件,并标注 `Note: file list is sampled`——保留"这个技能有哪些文件"的结构信息,但不把几十个文件全塞进上下文。第 8 章工具输出的 `TOOL_OUTPUT_MAX_CHARS` 上限也是同一原则:留住输出的开头(结构与要点),截断膨胀的尾部。

**怎么借走**:每当你要丢弃数据(为了省 token、为了脱敏、为了导出),先问:这份数据的"形状"对后续理解有没有价值?如果有,设计一个保留形状、压缩内容的表示——固定模板的摘要、采样的清单、带计数的占位符——而不是整块删除。

## 原则六:不变式固化进类型和测试(Invariants in types & tests)

**关键契约不靠注释维系,而靠编译期类型 + 运行期断言强制。**

这是贯穿全书最密的一条线。OpenCode 几乎不靠"开发者记得"来维持正确性,它把不变式钉进两个不会撒谎的地方:类型和测试。

类型层面:第 12 章的事件表用一个 `uniqueIndex(aggregate_id, seq)` 在数据库 schema 层面强制"同一聚合内序号不重复"——这是乐观并发控制的物理基石,写进了表结构而非代码注释。第 13 章 MCP 配置的 `callback_port` 用 `Schema.isBetween({minimum:1, maximum:65535})` 把端口约束进合法区间,非法值在解码时就被挡掉。第 7 章把五家 provider 的差异收敛进统一的类型契约,差异被类型吸收而非散落在 if 分支里。

测试层面:第 15 章整章都是证据。权限的每一条规则——三效果立即返回、空规则集兜底为 ask、fail-closed deny、saved 只升级不覆盖、ask 解决一次——都被写成 `expect(...).toEqual(...)` 的精确断言。`Effect.flip` + `toBeInstanceOf(DeniedError)` 把"这里必须失败成这个错误"钉死。第 12 章的事件重放幂等性靠 `isDeepStrictEqual` 强制,第 15 章的工具测试用精确的结果匹配守护工具行为。

**怎么借走**:对每一条你不想被破坏的契约,问:它现在靠什么维系?如果答案是"注释"或"约定",就把它上移——能进类型系统的进类型(字面量类型、范围约束、唯一索引、标签联合),进不了类型的进测试(精确断言、错误类型断言)。注释会过时,类型和测试不会。

## 原则七:可信度即流水线(Trustworthiness as a pipeline)

**可信不是某个单点开关,而是依赖注入 + 隔离 + 断言 + 多层防御组成的一整条流水线。**

前面六条原则,最终在第七条里汇合。OpenCode 之所以可信,不是因为它在某处做对了一件事,而是因为它的整个架构是为可信而组织的。

依赖注入是这条流水线的脊椎。第 4 章的 Session 运行时、第 14 章的服务端、第 15 章的测试,都建立在 Effect 的 `Layer` + `Context.Service` 之上。因为依赖是注入的,第 15 章的测试才能把磁盘库换成 `:memory:`、把执行后端换成 `noopLayer`、把时钟换成 `TestClock`——被测代码一个字都不用改。第 14 章才能把执行后端 `SessionExecutionLocal` 作为可替换 Layer 注入,暗示实现可换而契约不变。

隔离是流水线的关节。第 13 章每个插件活在独立的 `Scope` 里,失败即回收、卸载即清理;第 15 章每个测试 `Effect.scoped` 自动回收资源、用独立内存库互不污染、临时目录 `Symbol.asyncDispose` 自动清理还带 EBUSY 重试。

多层防御是流水线的冗余。第 13 章的路径逃逸防御叠了四层,第 10 章的权限叠了 agent 规则 + saved 规则 + fail-closed 默认,第 14 章的认证叠了 required 判定 + 凭据校验 + Redacted 遮蔽。没有任何单点失效能击穿整条链。

断言是流水线的质检站,已在原则六详述。

这四者——注入、隔离、断言、多层防御——共同把"我觉得它没坏"变成"我能证明它没坏"。对一个本质上不确定的 AI 系统,这是唯一可靠的可信度来源。

**怎么借走**:不要把可信度寄托在某一个安全检查上。把它建成流水线:用依赖注入让一切可替换、可测试;用作用域隔离让失败不扩散;用多层独立防御让单点失效不致命;用断言守护每一条不变式。可信不是一个功能,是一种架构姿态。

## 尾声:框架会过时,判断不会

OpenCode 用了 Effect,而 Effect 本身相当小众。如果你只从这本书里学到"怎么用 Effect 写 agent",那这本书几年后就过时了。

但如果你学到的是上面这七条——不确定时倒向安全、约束只收紧、把错误设计成文档、在每道边界上不信任外部输入、丢数据时留住形状、把不变式钉进类型和测试、把可信度当成一整条流水线来建——那么无论框架怎么变,无论你下一个 agent 用什么语言,这些判断都还成立。

这正是解构一个优秀代码库的意义:我们读的是 TypeScript,学的是判断力。OpenCode 在每一个决策点上展现的那种"宁可受限也不冒险、宁可显式也不隐式、宁可可证也不自信"的取向,才是它作为一个生产级 agent 真正的护城河。把这种取向内化成你自己的默认反应,你就不再需要这本书了——而那,恰恰是这本书最想达到的结局。

## 本章小结

- 本章不引入新代码,而是把全书观察提炼成七条可迁移的工程原则,每条都回指前面章节作为证据。
- **默认即关闭**:不确定时倒向受限一侧——第 10 章权限默认 ask、无 agent 即全 deny,第 14 章认证设密码即强制。
- **约束只收紧**:即便 v2 权限是叠加而非交集,也用显式 deny + deny-wins 求值补上收紧保证——第 10、11、13 章的子 agent 全-deny 起步。
- **错误即文档**:错误是带标签的类型化值,可被 `catchTag` 精确翻译成 HTTP 语义;内部错误对外给引用号、对内留全貌——第 7、8、12、14 章。
- **边界不信任**:每道信任边界都设校验闸,远程技能下载叠四重路径逃逸防御并"解码后重校验"——第 9、12、13、14 章。
- **留结构去内容**:压缩成固定模板摘要、采样文件清单、输出截断上限,保住形状抹掉冗余——第 8、11、13 章。
- **不变式入类型与测试**:唯一索引、范围约束、标签联合写进类型,权限规则与重放幂等写成精确断言——第 7、12、15 章。
- **可信度即流水线**:依赖注入(可替换/可测)+ 作用域隔离(失败不扩散)+ 多层防御(单点不致命)+ 断言(守护不变式)四者合一——第 4、13、14、15 章。
- 终极结论:框架会过时,判断不会。解构优秀代码库,读的是实现,学的是判断力。

## 动手实验

1. **给自己的 agent 做一次 fail-closed 审计**:挑出你(或你正在读的另一个 agent)代码里所有的权限/配置 fallback 分支和默认值,逐个判断它倒向"能做"还是"不能做"。把所有倒向"能做"的列成清单,逐条论证它是否真的安全。这是把原则一变成肌肉记忆的最快方式。

2. **复刻四重路径逃逸防御**:用任意语言实现一个"从远程 index 下载文件到本地目录"的最小函数,并加上第 13 章那四层防御(字符串校验、解码后重校验、解析后路径包含检查、同源检查)。然后写几个攻击性输入(`../`、`%2e%2e`、绝对路径、跨域 URL)验证每一层各挡住了什么。体会单层防御为何不够。

3. **把一条注释契约升级成类型或测试**:在你自己的项目里找一条目前只靠注释维系的不变式(比如"这个 id 必须是 48 位十六进制""这两个数组长度必须相等")。参照原则六,把它上移——能进类型就用字面量类型/范围约束/唯一索引表达,进不了就写一条精确断言测试。对比升级前后,这条契约被破坏时反馈来得快了多少。

4. **画一张你 agent 的可信度流水线图**:参照原则七,画出你系统里的四个环节——哪些依赖是注入的(可替换/可测)、哪些边界做了隔离(作用域/进程/事务)、哪些地方叠了多层防御、哪些不变式有断言守护。找出图上最薄弱的一环(通常是"某个安全检查是单点的、没有第二层"),设计一个补强方案。这张图就是你系统可信度的体检报告。
