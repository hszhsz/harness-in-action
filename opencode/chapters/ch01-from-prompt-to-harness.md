# 第 1 章　从 Prompt 到 Harness：Agent 工程范式的演进

> 如果你只是想让模型"回答一个问题"，一次 API 调用就够了。
> 但如果你想让它"自己读代码、自己改文件、自己跑命令、自己验证结果"——你需要的就不再是一个 Prompt，而是一整套 **Harness**。

本章不急着进入 OpenCode 的源码。我们先把一个更根本的问题讲清楚：**为什么一个生产级 Agent 的复杂度，绝大部分都不在模型，而在模型之外的工程系统里。** 想清楚了这件事，你才能理解后面 15 章我们到底在拆什么、为什么值得拆。

## 1.1 `Agent = Model + Harness`

把今天的 AI 编码 Agent 拆开，你会发现它由两部分组成：

- **Model（模型）**：负责"想"。给它一段上下文，它吐出文本、思考过程、以及"我要调用哪个工具、参数是什么"的结构化意图。
- **Harness（驾驭层）**：负责"做"和"管"。它决定模型每一轮能看到什么上下文、能调用哪些工具、工具执行要不要先经过人类批准、对话状态存在哪里、出错了怎么恢复、跑长了怎么压缩。

模型是买来的、可替换的、各家差不多的；Harness 才是一个 Agent 产品真正的护城河和工程难点。OpenCode 这样的项目，**几乎全部代码都是 Harness**——它本身不训练任何模型，而是把 Anthropic、OpenAI、Google、Bedrock 等家的模型，通过一套统一的协议适配层接进来（我们会在第 7 章看到 `packages/llm/src/protocols` 下并列摆着 `anthropic-messages.ts`、`openai-chat.ts`、`openai-responses.ts`、`gemini.ts`、`bedrock-converse.ts` 五套协议实现）。

这意味着：**理解一个 Agent，本质上就是理解它的 Harness。** 模型层是一个可以换的"发动机"，而 Harness 是底盘、变速箱、刹车、仪表盘和安全气囊。

## 1.2 为什么同一个模型，在不同产品里表现天差地别

你一定有过这种体验：同样是 Claude，套在一个玩具脚本里只会胡乱编造文件路径，套在一个成熟的编码 Agent 里却能精准地定位文件、做最小化修改、跑测试验证。模型是同一个，差别全在 Harness。具体差在哪？至少这几处：

- **它能看到什么**：玩具脚本可能把整个仓库一股脑塞进上下文，成熟 Agent 则有一套"按需装配"的上下文系统。OpenCode 的 v2 内核甚至为此专门写了一份领域语言定义文档 `CONTEXT.md`，把"模型看到的上下文"拆成 **System Context**、**Session History**、**Context Source**、**Context Epoch** 等十几个有精确定义的概念——光是"模型该看到什么"这一件事，就值得一整套领域建模。
- **它的工具有多可靠**：模型给出的工具参数经常不合 schema、路径越界、命令危险。成熟 Agent 在工具执行前有校验、有权限闸门、有沙箱。
- **它跑长了会不会崩**：长任务会撑爆上下文窗口、会让 OAuth token 过期、会累积无关历史。成熟 Agent 有压缩（compaction）、有重试退避、有状态持久化。

这些"差别"，每一条都是后面某一章的主题。**本书的价值，就是把这些让模型"变可靠"的工程机制，一条条从 OpenCode 的真实代码里挖出来。**

## 1.3 Harness 的核心职责

综观 OpenCode 的目录结构（下一章详述），一个生产级 Agent 的 Harness 大致承担六类职责。我们先建立这个心智地图，后面每一章都是在深入其中一格：

| 职责 | 解决的问题 | 本书对应章节 |
|---|---|---|
| **编排（Orchestration）** | 怎么把"生成→调工具→喂回结果"串成一个能自主推进的循环 | 第 4、5 章 |
| **上下文（Context）** | 每一轮到底把什么放进模型的视野 | 第 6、11 章 |
| **工具（Tools）** | 怎么定义、注册、调度一批能改变世界的能力 | 第 8、9 章 |
| **约束（Constraints）** | 怎么在动手前踩刹车，拦下危险操作 | 第 10 章 |
| **状态（State）** | 对话历史、会话、轨迹存在哪、怎么恢复 | 第 12 章 |
| **接入与扩展（IO & Extension）** | 怎么对接多种模型、多个前端、外部工具 | 第 7、13、14 章 |

注意这六类职责之间是有依赖顺序的：没有可靠的编排，上下文工程无从谈起；没有受约束的工具，状态持久化只会忠实地记录灾难。所以本书的章节顺序不是随意的，而是**顺着一次真实请求穿过这些子系统的路径**来排的。

## 1.4 本书方法论：以真实代码为唯一真理来源

市面上讲 Agent 的材料，最大的问题是**脑补**——用"Agent 一般都这么做"的常识，去描述一个其实没读过源码的机制。本书刻意反着来，只守一条铁律：

> **每一个技术论断，都必须能在 OpenCode 的真实源码里找到对应。**

具体来说：

- 我们引用**确切的文件名**（如 `packages/core/src/session/runner/index.ts`）、**确切的函数名/类名**（如 `StepLimitExceededError`）、**确切的字面量与默认值**。
- 写到某个机制之前，先真的把那个文件读一遍；拿不准就回去再读，绝不猜。
- 行内引用 `[[标题]](链接)` 只留给**外部可核查的事实来源**（官方规范、协议文档）。从 OpenCode 仓库里"读出来的"事实，直接写文件名即可，那不是外部引用。

为什么这么较真？因为 Agent 工程里魔鬼全在细节：一个默认值是 `true` 还是 `false`、一个合并是取交集还是并集、一个错误是抛异常还是被建模成正常结果——这些细节决定了系统是"可信"还是"能跑"。脑补会精准地跳过所有这些细节。

OpenCode 还给了我们一个额外的礼物：它**正处在 v1 到 v2 的架构迁移中**。`packages/opencode/src` 是仍在服役的 v1 运行时，`packages/core/src` 是基于 Effect 重写的 v2 内核。同一批工程问题，在两套范式下有两种解法摆在一起。这种"对照组"在大多数开源项目里是看不到的——它让我们不仅能看到"答案"，还能看到"答案是怎么演化的"。

> **给读者的建议**：本书每章末尾都有"动手实验"。Agent 工程是一门手艺，光读不练等于没读。强烈建议你 clone 仓库，边读边在真实代码里验证每一个结论。

## 本章小结

- 一个 Agent 可以拆成 `Model + Harness`：模型负责"想"，Harness 负责"做"和"管"；OpenCode 几乎全部代码都是 Harness。
- 同一个模型在不同产品里表现天差地别，差别全在 Harness——它决定模型看到什么、工具多可靠、跑长了会不会崩。
- Harness 有六类核心职责：编排、上下文、工具、约束、状态、接入与扩展，它们之间有依赖顺序，本书章节顺序即依此排布。
- OpenCode 用一整份 `CONTEXT.md` 领域语言文档来定义"模型看到的上下文"，说明"模型该看到什么"本身就是一个值得严肃建模的工程问题。
- 本书的唯一方法论是"以真实代码为唯一真理来源"：引用确切的文件、函数、字面量，拒绝脑补。
- OpenCode 处于 v1（`packages/opencode/src`）到 v2（`packages/core/src`）的迁移中，给了我们一个罕见的"同一问题两种解法"的对照组。

## 动手实验

1. **实验一：建立你的代码地图** — clone `https://github.com/anomalyco/opencode.git`，在仓库根目录运行 `ls packages`，对照本章 1.3 的六类职责，猜一猜每个包大概对应哪一格。先猜，下一章我们对答案。
2. **实验二：找到"两套实现"** — 分别看 `packages/opencode/src/session` 和 `packages/core/src/session` 两个目录下的文件列表。它们都叫 session，但文件名差异很大。记下你觉得"明显是 v2 新增的概念"的文件名（提示：`context-epoch.ts`、`projector.ts`、`run-coordinator.ts`）。
3. **实验三：读领域语言** — 打开仓库根目录的 `CONTEXT.md`，只读"Language"一节。数一数它为"上下文"这一件事定义了多少个专有名词。思考：为什么一个编码 Agent 需要给上下文做这么细的领域建模？
4. **实验四：确认作者的工程纪律** — 打开 `AGENTS.md`，读"Style Guide"一节。记下三条你认同的工程约束（例如"Avoid `else` statements. Prefer early returns"）。带着这些约束去读后面的源码，你会发现它们被相当一致地执行了——这本身就是"可信工程"的一个信号。
