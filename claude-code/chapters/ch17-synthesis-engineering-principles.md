# 第 17 章　收尾综合：从 CCB 提炼可迁移的 Harness 工程原则

> 前面十六章，我们从进程入口一路拆到 Hooks 与 Plugin——主循环怎么跳动、工具怎么调度、权限怎么踩刹车、Bash 怎么过 AST 安全门、上下文怎么瘦身、记忆怎么召回、子 agent 怎么委派、外部能力怎么引入。每一章都是一组**具体的源码事实**:某个常量取了某个值、某个分支走了某条路、某段注释记了某次事故。这最后一章不引入任何新代码,而是退后一步问:这些散落在 CCB 各处的设计决策,能不能收敛成一套**与语言、框架无关、可迁移到任何 agent 项目**的工程方法论?答案是七条原则。它们不是事后套上去的模板,而是从前面读到的代码里**长出来**的——每一条都回指到前面具体章节的证据。如果说前十六章是「CCB 怎么做的」,这一章是「为什么这么做是对的,以及你下次造 harness 时该带走什么」。

## 17.1 原则一:默认即关闭(Fail-closed defaults)

**不确定时,选择更安全、更受限的一侧。** 这是贯穿全书出现频率最高的一条,因为 harness 的本质是「在模型这个不完全可控的执行体外面套一层约束」,而约束的默认值决定了「当配置缺失、输入异常、状态未知时」系统倒向哪一边。CCB 的回答始终是:倒向安全。

证据散落在每一层:

- **工具层**(第 6 章):统一工具契约的默认是 fail-closed——一个工具没显式声明它在某权限模式下可用,就当它不可用。约束的缺省分支走拒绝而非放行。
- **权限层**(第 8 章):动手前先踩刹车,审批流的缺省是「问」而非「直接干」。
- **Skill 属性**(第 15 章):`skillHasOnlySafeProperties` 是 **allowlist 而非 blocklist**——「将来给 skill 加的任何新属性,默认都要确认」。未知能力默认收紧,而不是默认放行等出事再补。
- **分发层**(第 16 章):`strictPluginOnlyCustomization` 让管理员一键把 skills/agents/hooks/mcp 四个面锁死到只信任 plugin 和 managed 源;HTTP hook 的 SSRF 守卫默认封掉所有私网段。

这条原则的可迁移内核是:**默认值不是「方便」的选择,是「出事时谁来兜底」的选择。** 当你写下一个 `DEFAULT_*` 常量、一个 switch 的 fallback 分支、一个配置 schema 的缺省时,问自己:如果这个值在最坏情况下生效,系统会更安全还是更危险?选更安全的那个。

## 17.2 原则二:约束只能收紧(Constraints only tighten)

**合并多来源的配置或权限时,结果只会比任何单一来源更严,绝不更松。** 这条原则在 CCB 里有一个特别清晰的复现链条——子 agent 的工具三层裁剪(第 14 章):

> 第一层 harness 划死不可逾越的红线(`ALL_AGENT_DISALLOWED_TOOLS`)、第二层按执行模式收紧(异步 agent 从黑名单切到白名单 `ASYNC_AGENT_ALLOWED_TOOLS`)、第三层让每个 agent 把范围再收窄到任务所需(`resolveAgentTools`)。三层叠起来,**只会让工具变少,不会变多**。

这种「每一层只能减、不能加」的单调性,正是约束收紧原则的几何形态。它还出现在:

- **记忆分层**(第 13 章):CLAUDE.md 四层(Managed→User→Project→Local)合并,越具体越贴近当下的层优先级越高,但任何一层都是在前面基础上**叠加约束**。
- **fork 路径的补闸**(第 14 章):fork 为了缓存一致绕过了第一层裁剪,CCB 没有因此放松,而是用 `filterParentToolsForFork` 在旁路**手动补回**同一道禁用清单——「为性能开了特例,就必须为特例补回安全闸」。约束被绕过时不是默许,而是另寻一处把它加回来。

可迁移内核:**当你设计配置合并或权限叠加时,让操作是「取交集」而非「取并集」,是 AND 而非 OR。** 任何一个能让合并结果比某个输入源更宽松的路径,都是一个潜在的提权漏洞。

## 17.3 原则三:错误即文档(Errors as documentation)

**错误信息本身要教会读者正确用法,而不只是抛一段栈。** CCB 把这条贯彻到两个层面:一是抛错时带可操作的修复指引,二是把**已知限制坦诚写进注释当文档**,而不是假装它不存在或搞一套复杂的规避。

- **诚实标注裂缝**(第 15 章):`mcpInfoFromString` 的注释直说「如果 server 名含 `__`,解析会出错」——CCB 没有强行修复,而是把限制写进注释并说明「rare in practice」,靠 normalize 把常见情况兜住。
- **SSRF 守卫的已知绕过**(第 16 章):`ssrfGuard` 注释明说「代理在用时这个守卫对目标 host 实质上被绕过,由沙箱域名 allowlist 兜底」——把防御的边界明明白白交代清楚。
- **未知输入报错而非静默吞**(第 16 章):`processHookJSONOutput` 遇到未知的 `decision` 或 `permissionDecision` 类型一律 `throw`,错误信息直接列出「Valid types are: approve, block」——告诉你正确的取值。

可迁移内核:**错误信息是给「正处于困惑中的人」的文档,这是它唯一被阅读的时刻。** 抛错时带上字段名、期望格式、修复建议;遇到已知限制,把它写成注释而非埋起来——后来的维护者会因为这份坦诚少踩一个坑。

## 17.4 原则四:边界处不信任外部输入(Boundary distrust)

**任何跨越信任边界进入系统的数据,都先校验、净化、或重新解析。** Harness 的信任边界格外多:用户输入、模型输出、工具结果、MCP server 推来的内容、plugin/marketplace、文件路径。CCB 在每一道边界上都设了不信任的关卡。

- **Bash 安全门**(第 9 章):命令不是字符串拼接执行,而是先 AST 解析、做只读判定、过沙箱——把「一条命令字符串」当成不可信输入彻底拆解。
- **本地 vs 远程的区别对待**(第 15 章):本地 skill 的 markdown 是用户自己写的(可信),可以执行内联 bash;MCP server 推来的 markdown 是远程的(不可信),`loadedFrom==='mcp'` 时**绝不执行内联 shell**。同样的语法,按来源的可信度被区别对待。
- **同形字防伪**(第 16 章):marketplace 名只允许 ASCII,`NON_ASCII_PATTERN` 堵死用西里尔字母冒充 `anthropic` 的钓鱼;保留名必须来自 `anthropics/` org。
- **命名 normalize 与去重**(第 15 章):MCP server 名过 `normalizeNameForMCP` 把非法字符换下划线;skill 去重用 `realpath` 而非不可靠的 inode。

可迁移内核:**画出你系统的信任边界,在每一条边界上假设进来的数据是恶意的。** 路径要防逃逸、URL 要防 SSRF、名字要防同形字、序列化要防原型污染。可信的内部数据和不可信的外部数据即便长得一样,也要走不同的处理路径。

## 17.5 原则五:留结构、去内容(Preserve structure, remove content)

**当你必须删减数据时(脱敏、压缩、截断),保留它的形状,抹掉它的负载。** 这条原则让「减少」变得可逆、可诊断,而不是粗暴地一删了之。

- **上下文瘦身三段**(第 11 章):预算、snip、microcompact 都「不读懂对话讲了什么」,只按字节、UUID、tool_use_id 机械搬运——保留消息的结构骨架,只挪走占空间的内容。
- **autocompact 的九段模板**(第 12 章):即便是破坏式摘要,也被钉成固定九段结构,强制覆盖「意图/技术/文件/错误/用户原话/待办/现场/下一步」;第 6 段「列出所有用户消息」、第 9 段「逐字引用」专防摘要漂移——结构是防信息丢失的脚手架。
- **HTML 注释剥离的克制**(第 13 章):`stripHtmlComments` 删注释,但保留代码块内的 `<!-- -->`(可能是正在讲解的示例)、未闭合注释原样留——「拿不准就不动」。
- **skill 按头部截断**(第 12 章):压缩后恢复 skill 时「按 skill 截断保头部」而非整体丢弃,因为「skill 文件开头的说明通常最关键」。

可迁移内核:**删减时优先删「内容」保「结构」**。一个保留了对象形状、只把值换成掩码的脱敏日志,比一个整段消失的日志有用得多——前者还能诊断,后者只剩黑洞。

## 17.6 原则六:不变量固化进类型与测试(Invariants in types & tests)

**关键契约不靠注释提醒,而靠编译期类型和运行期断言强制。** 注释会过时、会被忽略,但类型错误编译不过、断言失败测试变红。CCB 把大量安全属性钉进了枚举、字面量类型和守卫函数。

- **封闭枚举当契约**:记忆类型被钉成封闭四类(第 13 章 `MEMORY_TYPES`)、hook 事件被钉成 27 个 `HOOK_EVENTS`(第 16 章)——枚举之外的值根本无法通过 schema 校验,写错的事件名在配置加载时就被 Zod 拦下,而非静默不触发。
- **死循环守卫写成状态字段**(第 12、16 章):`hasAttemptedReactiveCompact`、`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES=3`、`MAX_PTL_RETRIES=3` 把「不许无限重试」这个不变量固化成可检查的计数器和布尔标志,而不是寄望于代码逻辑「应该不会循环」。
- **schema 即契约的反面教训**(第 16 章):`AgentHookSchema` 上那段「绝不能加 `.transform()`」的注释,记录的正是一次「类型契约没守住导致用户数据被静默丢弃」的事故(gh-24920)——契约一旦破,损失是静默的。

可迁移内核:**把「绝不能发生」的事写成类型不会让它编译、写成断言会让测试变红。** 一个 `as const` 枚举、一个 `MAX_*` 上限常量、一个 fail-loud 的校验分支,胜过十行「请注意不要……」的注释。

## 17.7 原则七:可信度是一条流水线,不是一个开关(Trustworthiness as a pipeline)

**「可信」不是某处打个勾就完成的,而是多层防御、隔离、对账、缓存一致性共同织成的流水线。** 这是最综合的一条,它把前面六条串起来——任何一处的可信都依赖上下游一连串机制的协同。

- **缓存一致性偏执的三次复现**(第 12、13、15 章):fork 不设 maxOutputTokens(防 thinking 配置不一致 cache miss)、age 字符串预计算(防 `Date.now()` 漂移 cache bust)、MCP instructions 按 server NAME diff 产出增量 attachment(防晚连接 server 每 turn 重拼系统提示)——三处看似无关的设计,共享同一条不变量:**任何会进 API 请求前缀的内容,只要字节可能随时间或状态变动,就必须「一次确定、之后不变」。** 可信的性能不是单点优化,是一条贯穿多个模块的纪律。
- **多层防御叠加**:子 agent 的工具裁剪是三层(第 14 章)、死亡螺旋的防护是三道堵口(第 12、16 章)、压缩动作自身的保护是「先给摘要留预算 + 防图片撑爆 + 连续失败熔断」多重(第 12 章)。没有任何单点能独自保证安全,是叠加产生了鲁棒性。
- **生命周期收尾的不变量**(第 14、15 章):子 agent 无论成功/失败/abort,`finally` 都要清任务注册、清 `clearInvokedSkillsForAgent`、清 dumpState、按需保留 worktree——可信度要求「无论从哪条路径退出,状态都被正确收束」。
- **对账机制**(第 12 章):snip 删了消息但 usage 仍反映删前状态,所以触发判断要手动减去 `snipTokensFreed`——不同瘦身阶段之间要对账,可信度要求各子系统对同一份状态的认知一致。

可迁移内核:**不要指望用一个「安全模式」开关获得可信度。** 可信是 DI 便于替换测试、是测试隔离基建、是多层校验链、是缓存 key 逐字节对齐、是每条退出路径都收尾干净——它是一种渗透到每个模块的工程纪律,而非一个功能特性。

## 本章小结

- 本章不引入新代码,把前十六章的观察提炼成七条与语言/框架无关、可迁移的 harness 工程原则,每条回指具体章节作为证据。
- **原则一·默认即关闭**:不确定时倒向安全/受限的一侧(第 6 章工具 fail-closed、第 8 章审批默认问、第 15 章 skill allowlist、第 16 章 plugin-only 锁)。默认值是「出事时谁兜底」的选择。
- **原则二·约束只能收紧**:多来源合并结果只严不松(第 14 章子 agent 三层裁剪只减不增、第 13 章记忆分层叠加、第 14 章 fork 绕过后补回安全闸)。操作取交集/AND 而非并集/OR。
- **原则三·错误即文档**:错误信息教正确用法、已知限制坦诚写进注释(第 15 章 `mcp__` 解析裂缝、第 16 章 SSRF 已知绕过、未知输入 `throw` 列出合法值)。
- **原则四·边界不信任**:跨信任边界的数据先校验净化(第 9 章 Bash AST 安全门、第 15 章本地 vs MCP 区别对待、第 16 章同形字防伪、realpath/normalize)。可信与不可信数据走不同路径。
- **原则五·留结构去内容**:删减时保形状抹负载(第 11 章机械搬运不读语义、第 12 章九段模板防漂移、第 13 章注释剥离保代码块、skill 截断保头部)。让减少可逆可诊断。
- **原则六·不变量进类型与测试**:契约靠编译期类型 + 运行期断言强制而非注释(第 13/16 章封闭枚举、第 12/16 章死循环计数器守卫、`AgentHookSchema` 禁 transform 的反面教训)。
- **原则七·可信度是流水线**:可信由多层防御 + 隔离 + 对账 + 缓存一致性协同织成(第 12/13/15 章缓存偏执三次复现、多层防御叠加、`finally` 收尾不变量、snip 对账)。不是单点开关,是渗透各模块的纪律。
- 七条原则相互咬合:fail-closed 提供默认安全、约束收紧保证合并不提权、边界不信任挡住外部攻击、错误即文档让维护者少踩坑、留结构去内容让瘦身可逆、不变量进类型让契约不漂移、可信度流水线把前六条串成整体——这正是 CCB 作为一个可信 harness 的工程内核。

## 动手实验

1. **实验一:用七条透镜审计一个新模块** — 任选 CCB 里本书未详细展开的一个文件(如 `src/services/mcp/` 下某个未拆过的模块),通读它,逐条对照七原则:它在哪里体现了 fail-closed?有没有约束收紧?错误信息是否教用法?边界校验在哪?有没有留结构去内容?哪些不变量写进了类型/测试?把发现整理成一页「原则命中表」。
2. **实验二:找一处「原则缺席」并论证后果** — 在你自己或公司的某个 agent/脚本项目里,找一个**违反**某条原则的地方(如默认放行、配置合并取并集、错误只抛栈、外部 URL 不校验)。论证:如果遭遇最坏输入,会发生什么?照对应原则改造后,攻击面如何收窄?
3. **实验三:复盘「为特例补闸」模式** — 重读第 14 章 fork 路径的「绕过 + 补闸」(`filterParentToolsForFork`)和第 16 章 plugin 绕过用户源后只放行 `ADMIN_TRUSTED_SOURCES`。论证:这两处都是「为某个目标(缓存一致/分发便利)开了特例,又为特例补回安全闸」。在你的系统里,有没有「为性能/便利开了特例却忘了补闸」的地方?
4. **实验四:提炼你自己的第八条原则** — 七条原则提炼自 CCB,但任何一个成熟 agent 都可能呈现 CCB 没有、或本书没强调的工程模式(如可观测性脱敏、回滚/幂等、限流退避)。从你读过的代码里**长出**一条本书未列的原则,给出 2–3 处源码证据,并说明它和这七条如何咬合或互补。

> **全书终**:从第 1 章「为什么要解构 Claude Code」到这里的七条工程原则,我们走完了一条「由外到内、由现象到原则」的路。CCB 不是一份完美的代码,它有诚实标注的裂缝、有记着事故编号的注释、有为性能开的特例;但正是这些**真实的工程权衡**,而非理想化的架构图,构成了一个可信 harness 的样貌。把这七条原则带走——下次当你给一个模型套上 harness 时,你要做的不是发明新魔法,而是在每一个默认值、每一道边界、每一次合并、每一处删减上,做出和 CCB 同一种「倒向安全、只收不放、坦诚边界、保结构、固化不变量」的选择。这,就是 harness 工程。
