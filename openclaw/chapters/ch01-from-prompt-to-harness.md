# 第 1 章　从 Prompt 到 Harness：Agent 工程范式的演进

> 「软件工程团队的主要工作不再是编写代码，而是设计环境、指定意图、并构建反馈循环，让 Agent 可靠地工作。」
> —— 这句话，是理解本书一切的起点。

如果你正在读这本书，多半已经写过不少和大模型打交道的代码：拼一段 Prompt，调一次 API，把返回的文本解析出来，再渲染给用户。它能跑，有时候效果还不错。但当你试图把它做成一个**能真正干活、能连续干活、能在出错后自己爬起来**的系统时，你会撞上一堵墙——这堵墙不在模型那边，而在模型之外的那一整套系统里。

这一章不写代码。它要先把一个问题讲透：**为什么"调好 Prompt"远远不够，以及我们到底在解构 OpenClaw 的什么。** 想清楚这件事，后面二十一章的每一行代码你才知道它"为什么存在"。

---

## 1.1 `Agent = Model + Harness`：能力边界由谁决定

我们先给出本书反复使用的一个等式：

> **Agent = Model + Harness**
>
> 等价地：**Harness = Agent − Model**

这个等式朴素得近乎废话，但它一旦被你真正接受，对 Agent 的认知会立刻清晰起来。

**Model（模型）**，是 OpenAI、Anthropic、Google 这些公司训练出来的那个东西。它的能力上限——推理有多强、知识有多广、上下文能装多长——**不由你决定**。你能换一个更强的模型，但你无法在工程层面把一个模型变得更聪明。

**Harness（驾驭层）**，是一个 Agent 环境中**除模型之外的所有东西**。它决定了：

- 模型**看到什么**（上下文：系统提示、历史对话、项目文档、工具返回值）
- 模型**能做什么**（工具：读写文件、执行命令、调用 API、操作浏览器）
- 模型**按什么规则做**（约束：权限边界、格式规范、危险操作拦截）
- 模型**做错了怎么纠偏**（反馈：运行测试、检验输出、重试与回滚）
- 最后**如何把能力稳定地交付出去**（编排与状态：分步执行、跨会话记忆）

"Harness" 这个词借自马具——缰绳、马鞍、辔头的总称。一匹马很强壮也很快，但没有驾驭系统，它只会随心所欲地跑。**模型就是那匹马，Harness 是引导它的力量产出成果的一切，而工程师，是那个提供方向的骑手。**

这就引出了一个关键的责任划分：

| | 由谁决定 | 工程师能否改变 |
|---|---|---|
| **模型能力上限** | OpenAI / Anthropic / Google | 否（只能换模型） |
| **能力的稳定落地** | Harness（你的工程） | **是，这正是你的主战场** |

本书要解构的 OpenClaw，本质上就是一套被打磨得相当成熟的 Harness。我们拆的不是某个模型有多强，而是**一群工程师如何用工程手段，把一个不确定的文本生成器，驯服成一个可以连续完成真实任务的系统**。

## 1.2 为什么同一个模型在不同产品里表现天差地别

接受了上面的等式，有两个困扰很多人的现象就立刻有了答案。

**现象一：为什么同样接 Claude，有的产品像"成熟员工"，有的却像"实习生"？**

因为它们的模型一样，但 Harness 天差地别。"实习生"那个，可能只是把用户输入直接拼进 Prompt 丢给模型，返回什么算什么；"成熟员工"那个，则在背后做了一整套工作：精心组织了上下文边界、给了模型合适的工具、规定了它出错时该怎么重试、在危险操作前踩了刹车。**同样的"大脑"，配上不同的"工作环境与流程"，产出的职业素养就完全不同。**

**现象二：为什么有时候"换一个更贵的模型"还不如"改一下系统提示"有用？**

因为在很多场景下，瓶颈根本不在模型的智商，而在于**它看到的信息是乱的、缺的、或者过载的**。一个再强的模型，如果你把十篇不相关的文档一股脑塞给它，它也会抓不住重点。

这里有一个常被引用的量化佐证：在 Terminal Bench 2.0 编程基准上，**仅改变 Harness 而完全不换模型**——只是加了一个自我验证循环和死循环检测——就把 Agent 的成功率从 52.8% 提升到了 66.5%[[What Is Harness Engineering? Complete Guide]](https://www.nxcode.io/resources/news/what-is-harness-engineering-complete-guide-2026)。

这个数字传递的信号很深刻：**在许多场景下，改进 Agent 的运行环境，比升级模型本身更有效。** 而运行环境，恰恰是工程师能 100% 掌控的部分。这也是为什么"做 Agent"在今天越来越不是一个调参问题，而是一个系统工程问题。

## 1.3 Harness 的六个核心职责

那么一套完整的 Harness 到底包含什么？业界有不同的拆法（有人拆五层，有人拆六层），本书采用一种对开发者最实用的六职责划分。它们也正是本书第二到第七部分的骨架。

| 核心职责 | 它解决的问题 | 在 OpenClaw 中的落点（后续章节） |
|---|---|---|
| **① 上下文管理** | 模型到底看到了什么 | 上下文引擎、System Prompt 构建、压缩剪枝（第 12–13 章） |
| **② 工具系统** | 模型到底能做什么 | 统一工具计划、内置编码工具（第 8–9 章） |
| **③ 执行编排** | 模型下一步该做什么 | Agent 主循环、Harness 编排层（第 4–5 章） |
| **④ 状态与记忆** | 系统如何跨步骤、跨会话保持连续 | SQLite 持久化、Session 恢复（第 14 章） |
| **⑤ 评估与观测** | 系统怎么知道自己做得对不对 | 测试分层、评估场景、架构守卫（第 21 章） |
| **⑥ 约束与恢复** | 出错了怎么办、怎么避免跑偏 | 权限审批、信任模型、安全审计（第 10–11 章） |

我们逐一用一句话点透每一项的本质：

**① 上下文管理——让模型在正确的信息边界内思考。** 它的反直觉之处在于：上下文不是越多越好。一个好的 Harness 既要把相关信息挑出来，也要把不相关的信息**挡在外面**，还要把它们结构化分层（固定规则 / 当前任务 / 运行状态 / 外部证据），显著降低模型"看漏重点"或"忘记约束"的概率。

**② 工具系统——把模型从"只会说"变成"能做事"。** 难点不在于"挂上工具"，而在于三个更深的问题：给它**什么**工具（写作 Agent 和安全分析 Agent 该有完全不同的工具集）、**什么时候**调用（既不能该查不查、也不能不该查乱查）、以及工具返回的结果**如何被提炼后再喂回**模型。

**③ 执行编排——让模型知道下一步该干什么。** 很多失败的 Agent 不是因为不会某一步，而是因为不会"串起来"，想到哪做到哪，最后输出一堆半成品。执行编排要明确地搭好轨道：步骤划分、决策节点、中间产物、终止条件、异常处理。

**④ 状态与记忆——让系统跨步骤、跨会话保持连续性。** 这里有 Agent 工程一个特别要命的现实：长时运行的 Agent 必须在**离散的会话**中工作，每个新会话开始时对之前发生的一切毫无记忆——就像一个软件项目由轮班工程师组成，每个新人到岗时对上一班一无所知。Harness 必须替它解决"交接班"。

**⑤ 评估与观测——让系统能回答"这次为什么成功/失败"。** 没有评估就没有优化闭环。你需要全链路日志、工具调用轨迹、关键指标看板、失败 case 留存，才能从"感觉更好了"变成"确定更好了"。

**⑥ 约束与恢复——既帮模型做事，也替模型兜底。** 这是"生产级 Harness"和"Demo"的分水岭。哪些操作需要人工确认？哪些数据不能泄露？外部写操作要不要二次确认？怎么防 Prompt 注入和权限绕过？

> 把这六句话记住。本书后面拆开 OpenClaw 的每一个目录、每一个文件，本质上都是在回答"它是如何实现这六个职责中的某一个"的。

## 1.4 范式的三次演进：我们今天站在哪里

把镜头拉远，会发现"和模型协作"这件事，经历了三次清晰的范式跃迁。

**第一代：Prompt Engineering（提示工程）。** 核心动作是"把话说好"——用更精巧的措辞、更清晰的指令、few-shot 示例，去激发模型的能力。它解决的是"单次对话质量"问题。但它的天花板很明显：模型说完就结束了，它不能接触真实世界，也记不住上一轮。

**第二代：Context Engineering（上下文工程）。** 核心动作从"说好一句话"升级为"管理好模型看到的全部信息"——检索增强（RAG）、记忆系统、上下文窗口的组织与裁剪。它解决的是"模型看到什么"的问题，是上一节六职责里的第①项。但它仍然主要关注"输入端"。

**第三代：Harness Engineering（驾驭工程）。** 核心动作扩展为"设计 Agent 运作的整个环境系统"——不只管输入，还管工具编排、安全护栏、反馈循环、可观测性、以及跨会话的状态管理。它把上下文工程包含在内，但把视野扩展到了整个执行闭环。这就是我们今天所处的范式，也是 OpenClaw 这类系统所代表的工程形态。

有一句对 Harness 工程精神最精炼的概括，值得你贴在工位上：

> **每当发现智能体犯了某个错误，就花时间通过工程手段给出一个解决方案，让它再也不会犯同样的错误。**

注意这句话的重音：不是"祈祷模型下次别犯错"，也不是"在 Prompt 里多写一句别犯错"，而是**用工程手段**——一段校验代码、一条 linter 规则、一个审批关卡——让那个错误在系统层面**不可能**再次发生。这正是工程师的本职，也是本书想教给你的思维方式。

## 1.5 本书方法论：以真实代码为唯一真理来源

市面上讲 Agent 的内容，大多停在概念层（像本章到目前为止这样）或 Demo 层（拼一个能跑的玩具）。本书要做一件更费力但也更有价值的事：**拿一个真实的、生产级的开源 Agent，把它彻底拆开。**

我们选的标本是 **OpenClaw**。

OpenClaw 对自己的定位是一句话：**"the AI that actually does things. It runs on your devices, in your channels, with your rules."**（一个真正能做事的 AI，运行在你的设备上、你的渠道里、按你的规则）[[OpenClaw VISION]](https://github.com/openclaw/openclaw/blob/main/VISION.md)。它从一个学习 AI 的个人项目起步，经历了 Warelay → Clawdbot → Moltbot → OpenClaw 的数次更名与重塑[[OpenClaw VISION]](https://github.com/openclaw/openclaw/blob/main/VISION.md)，演化为今天这样一个规模可观的系统。

选它做标本，有几个理由：

- **它是真·生产级，不是教学玩具。** 在本书写作时，它是一个 TypeScript 的 pnpm monorepo，包含 21 个可复用 package、142 个扩展（extensions）、58 个 Skill，版本号 2026.6.2。这个体量足以暴露出"真实 Agent 系统"才会遇到的全部工程难题。
- **它的工程纪律极强。** OpenClaw 有几条近乎严苛的"工程红线"——核心精简、能力插件化、配置只认当前 Schema（不为旧配置保留兼容分支）[[OpenClaw VISION]](https://github.com/openclaw/openclaw/blob/main/VISION.md)、状态只用 SQLite——并且用 CI 脚本强制守卫。这些约束本身就是绝佳的教材：每一条红线背后，都是一个值得你借鉴的架构决策。
- **它的优先级与 Harness 高度对齐。** OpenClaw 把"Security and safe defaults（安全与安全默认值）"列为第一优先级，并明确把"better computer-use and agent harness capabilities"写进发展方向[[OpenClaw VISION]](https://github.com/openclaw/openclaw/blob/main/VISION.md)。换句话说，它本身就是一个"以 Harness 为核心"的工程。

基于此，本书定下三条方法论原则：

1. **代码是唯一真理来源。** 本书的每一个论断都尽量落到具体的目录与文件路径上。概念会讲，但绝不悬空——你应该边读边 `git clone` 对照源码。当本书的描述与你手上的代码版本不一致时，**以代码为准**（开源项目演进很快）。

2. **顺着执行链路走，而不是按目录字母序走。** 我们不会枯燥地"从 a 目录讲到 z 目录"。本书的主线是**跟随一次真实请求穿过整个系统**：

   ```
   进程入口 (openclaw.mjs → entry.ts)
     → 运行时选择 (src/agents/runtime)
       → Agent 主循环 (packages/agent-core/agent-loop.ts)
         → 模型流式抽象 (llm-runtime / src/llm/providers)
           → 统一工具计划 (src/tools + agent-tools)
             → 权限审批与安全 (plugin-sdk/approval-*, src/security)
               → 上下文引擎与压缩 (context-engine, harness/compaction)
                 → 状态持久化 (SQLite, src/state)
                   → 扩展体系 (plugin-sdk / skills / MCP)
                     → 多端接入 (TUI / Web / Gateway / Channels)
   ```

   读完你脑中会有一张"数据如何在系统里流动"的动态地图，而不是一堆静态的孤立知识点。

3. **每拆一处，都回到"它实现了六职责中的哪一项"。** 这是贯穿全书的锚点。当你读到第 10 章的审批流时，要意识到这是职责⑥；读到第 13 章的压缩时，要意识到这是职责①。拆到最后，这六个职责会在你脑中长成一棵完整的树。

## 本章小结

- Agent 的能力 = 模型能力 + Harness 工程。模型的上限你改不了，但 Harness 是你的主战场。
- 同一个模型在不同产品里表现天差地别，差就差在 Harness；很多时候改 Harness 比换模型更有效。
- 一套完整的 Harness 承担六个核心职责：上下文管理、工具系统、执行编排、状态与记忆、评估与观测、约束与恢复。
- 我们正处在 Prompt → Context → Harness 三次范式演进的第三阶段：从"说好一句话"走到了"设计整个执行环境"。
- 本书以生产级开源项目 OpenClaw 为标本，以"真实代码为唯一真理来源"，顺着一次请求的执行链路，把上述六职责逐一拆开。

下一章，我们将拉一张 OpenClaw 的全景地图——在动手拆解任何一个子系统之前，先看清这套代码的整体疆域、它的设计哲学，以及那几条决定了它一切形态的"工程红线"。
