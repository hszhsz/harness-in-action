# 第 1 章　从 Prompt 到 Harness：编码 Agent 的工程范式

> 「模型决定了天花板，Harness 决定了你离天花板有多远。」
> —— 这是理解本书一切的起点。

如果你读到这本书，多半已经写过和大模型打交道的代码：拼一段 Prompt，调一次 API，把返回的文本解析出来。它能跑，有时候效果还不错。但当你想把它做成一个**能真正动手改代码、能连续干活、能在出错后自己爬起来**的系统时，你会撞上一堵墙——这堵墙不在模型那边，而在模型之外的那一整套系统里。

本书要拆的标本，是 OpenAI 官方开源的 **Codex CLI**。它不是一个 demo，而是一个用 Rust 写成、被打磨进生产的编码 Agent：它能在你的终端里读你的仓库、改你的文件、跑你的测试，并且在一台陌生机器上也敢把命令关进沙箱里执行。这一章不写代码，先把一个问题讲透：**为什么"调好 Prompt"远远不够，以及我们到底在解构 Codex 的什么。** 想清楚这件事，后面十六章的每一行 Rust 代码你才知道它"为什么存在"。

---

## 1.1 `Agent = Model + Harness`：能力边界由谁决定

我们先给出本书反复使用的一个等式：

> **Agent = Model + Harness**
>
> 等价地：**Harness = Agent − Model**

这个等式朴素得近乎废话，但一旦你真正接受它，对 Agent 的认知会立刻清晰。

**Model（模型）**，是 OpenAI 这类公司训练出来的那个东西。它的能力上限——推理有多强、知识有多广、上下文能装多长——**不由你决定**。你能换一个更强的模型，但你无法在工程层面把一个模型变得更聪明。

**Harness（驾驭层）**，是一个 Agent 系统里**除模型之外的所有东西**。在 Codex 里，它具体落实为：模型**看到什么**（System Prompt、`AGENTS.md`、历史会话、工具返回）、模型**能做什么**（读写文件、`apply_patch` 改代码、执行 shell 命令、调 MCP 工具）、模型**按什么规则做**（沙箱策略、执行策略、审批门控）、模型**做错了怎么纠偏**（把 stderr、退出码、补丁失败回灌给模型）、以及**如何把能力稳定交付**（线程、turn 生命周期、Rollout 持久化）。

"Harness" 这个词借自马具——缰绳、马鞍、辔头的总称。模型是那匹强壮的马，Harness 是引导它产出成果的一切，而工程师是那个提供方向的骑手。

| | 由谁决定 | 工程师能否改变 |
|---|---|---|
| **模型能力上限** | OpenAI 等模型厂 | 否（只能换模型） |
| **能力的稳定落地** | Harness（你的工程） | **是，这正是你的主战场** |

Codex CLI 本质上就是一套被打磨得相当成熟、且**特别强调安全**的 Harness。我们拆的不是某个模型有多强，而是**一群工程师如何用 Rust 把一个不确定的文本生成器，驯服成一个敢在你机器上动手、又不会失控的编码系统**。

## 1.2 编码 Agent 的特殊性：它要"动手"，所以风险更高

通用聊天 Agent 出错，最多是说错话。**编码 Agent 出错，可能是删掉你的文件、提交错误的 commit、或者在你的机器上跑一条 `rm -rf`。** 这就是为什么 Codex 把大量工程投入花在了"动手前踩刹车"上，而不是"让模型更会聊天"上。

这种差异在 Codex 的代码里随处可见。它不会让模型直接拼一个 shell 字符串去跑，而是把命令送进一套**执行策略**（`exec_policy.rs`）做判定，再决定是直接放行、要求审批、还是禁止。它甚至维护了一份"危险前缀建议"清单 `BANNED_PREFIX_SUGGESTIONS`，把 `python`、`bash`、`sudo` 这类能逃逸出沙箱意图的前缀单独标记出来。它不会让模型随手覆盖文件，而是要求模型走 `apply_patch` 这条受控的改代码通道，每一次改动都被 `TurnDiffTracker` 记成可审计的 diff。

更进一步，Codex 默认**不信任自己**——它假设模型可能产出危险命令，于是在 Linux 上用 Landlock + seccomp、在 macOS 上用 Seatbelt、在 Windows 上用受限令牌把执行环境关进笼子（第 8 章详解）。这是一种典型的"默认即关闭"工程姿态：不确定时，选择更受限的一侧。

所以读 Codex，你会发现它的复杂度不是花在"让 Agent 更聪明"，而是花在**"让一个会动手的 Agent 值得被信任"**。这正是生产级编码 Agent 与玩具 demo 的分水岭。

## 1.3 Harness 的核心职责：本书的骨架

那么一套完整的编码 Agent Harness 包含什么？本书采用一种对开发者最实用的职责划分，它们也正是本书各部分的骨架。

| 核心职责 | 它解决的问题 | 在 Codex 中的落点（后续章节） |
|---|---|---|
| **① 进程与编排** | 程序如何启动、一次 turn 怎么跑 | `arg0` 多态、`CodexThread` 主循环（第 3–4 章） |
| **② 模型接入** | 怎么跟模型稳定地流式对话 | `ModelClient`、Responses API（第 5 章） |
| **③ 工具系统** | 模型到底能做什么 | `ToolRouter` / `ToolRegistry`、`apply_patch`（第 6–7 章） |
| **④ 安全约束** | 会动手的 Agent 怎么不失控 | 多平台沙箱、执行策略与审批（第 8–9 章） |
| **⑤ 状态与配置** | 跨步骤、跨会话如何连续可复现 | 配置合并、认证、Rollout/SQLite（第 10–13 章） |
| **⑥ 扩展与运维** | 怎么长出新能力、怎么观测 | MCP/插件/Skill、OTEL 遥测（第 14–15 章） |

逐一点透本质：

**① 进程与编排——把"一次提问"变成"一段可控的执行流"。** Codex 的入口不是一个简单的 `main()`，而是一套 `arg0` 多态分发机制：同一个可执行文件，根据被调用时的名字，可以是 CLI、是无头 exec、是 Linux 沙箱、甚至是 `apply_patch` 工具本身。这种设计让"一个二进制干多件事"成为可能，也是理解整个进程模型的钥匙。

**② 模型接入——让与模型的对话变成可重试、可计量、可流式的工程组件。** Codex 用 `ModelClient` 封装了与 OpenAI Responses API 的交互，支持 WebSocket 预热、流式响应、压缩历史，把"调一次 API"升级成"管理一条有状态的对话流"。

**③ 工具系统——把模型从"只会说"变成"能改代码"。** 难点不在于挂上工具，而在于：所有工具（内置、MCP、Skill）如何被同一套 `ToolRouter` 一视同仁地调度，又如何被同一套审批与沙箱约束覆盖。

**④ 安全约束——这是编码 Agent 的灵魂。** 沙箱、执行策略、审批流三者叠加，构成多层防御。任一层不是开关，而是流水线上的一道闸门。

**⑤ 状态与配置——让 Agent 可复现。** 多层配置合并遵循"约束只能收紧"，会话被记成 Rollout（JSONL）并落进 SQLite 状态库，使得任何一次运行都能被恢复、导出、审计。

**⑥ 扩展与运维——让 Agent 长出能力、并且看得见自己。** MCP / 插件 / Skill 是能力外挂的三种形态，OTEL 遥测与诊断导出则保证系统在生产里可观测、可排障。

## 1.4 本书方法论：以真实代码为唯一真理来源

本书只有一条铁律：**每一个技术论断，都来自我们实际读过的 Codex 源码文件。** 我们不写"编码 Agent 一般怎么做"的常识，只写"Codex 在 `core/src/exec_policy.rs` 里**确实**这么做了"。

这意味着你会在书里频繁看到确切的文件名（如 `codex_thread.rs`）、确切的结构体名（如 `TurnContext`、`ModelClient`）、确切的常量值（如 `TOKIO_WORKER_STACK_SIZE_BYTES = 16 * 1024 * 1024`）。这些不是为了炫技，而是为了让你能**亲自去仓库里核对每一句话**。本书的每一章末尾都附有"动手实验"，引导你打开对应的源码，自己验证我们讲的机制是否属实。

读这本书最好的姿势，是把 Codex 仓库 clone 下来，左边放代码、右边放书，边读边跳转。当你能在源码里找到我们引用的每一个名字时，你对这个系统的理解就不再是"听说"，而是"亲眼所见"。

## 本章小结

- **Agent = Model + Harness**：模型能力是天花板（你换不动），Harness 是你能 100% 掌控的工程主战场。
- **编码 Agent 的特殊性在于它要"动手"**：出错代价远高于聊天 Agent，所以 Codex 的工程重心是"让会动手的 Agent 值得信任"，而非"让 Agent 更会聊"。
- Codex 用 Rust 实现，把安全做成多层防御：执行策略（`exec_policy.rs`、`BANNED_PREFIX_SUGGESTIONS`）+ 受控改代码（`apply_patch` + `TurnDiffTracker`）+ 多平台沙箱（Landlock/seccomp/Seatbelt/受限令牌）。
- Harness 的六大职责构成本书骨架：进程与编排、模型接入、工具系统、安全约束、状态与配置、扩展与运维。
- 本书唯一方法论：**以真实源码为唯一真理来源**，每个论断都可在仓库里核对，每章末尾有动手实验。

## 动手实验

1. **实验一：建立你的阅读工作台** — 执行 `git clone https://github.com/openai/codex.git`，进入 `codex-rs/` 目录，运行 `ls` 观察这是一个 Rust workspace。把仓库在编辑器里打开，与本书并排放置，后续每章都对照源码阅读。
2. **实验二：找到"安全优先"的第一处证据** — 在 `codex-rs/core/src/exec_policy.rs` 中搜索 `BANNED_PREFIX_SUGGESTIONS`，看看 Codex 把哪些命令前缀标记为危险。思考：为什么 `python`、`bash` 会被单独列出？
3. **实验三：感受"动手"与"聊天"的差异** — 阅读 `codex-rs/core/src/turn_diff_tracker.rs` 的开头注释与结构体 `TurnDiffTracker`，理解 Codex 为什么要把每一次文件改动都记成可审计的 diff，而不是直接覆盖文件。
4. **实验四：数一数 Harness 有多大** — 在 `codex-rs/` 下运行 `ls -d */ | wc -l`，数出这个 workspace 有多少个 crate。对比"模型"只是其中很小一部分（接入逻辑在 `model-provider/`），直观感受"Harness ≫ Model"。
